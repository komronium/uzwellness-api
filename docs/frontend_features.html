UzWellness API — Frontend Guide

How the backend is organized and how to consume it from uzwellness.com. Base URL: https://api.uzwellness.com/api/v1.

1. Overview

Two product surfaces share one backend:

SANATORIUM

Medical resorts (Humson Buloq, Charvak Health…). The unit of sale is a room-night. Optional treatment programs come bundled with the stay based on its length.

WELLNESS

Yoga studios, spa resorts, meditation centers, detox retreats. The unit of sale is a program/session (drop-in class, weekend retreat) with its own price.

Same DB table, different property_type + a small set of discriminator columns. Both share images, reviews, amenities, location, and RBAC.

3. Room ↔ Program relationship

No foreign key directly connects rooms and programs. Both are siblings under a sanatorium: 

    sanatoriums
        │
        ├── room_categories   ← xona-tun sotiladi  (sanatorium uchun)
        └── treatment_programs ← xizmatlar paketi   (har 2 turi uchun)

They relate implicitly through stay length:

Sanatorium booking

  1. User picks a room + check-in/out (e.g. 7 nights)
  2. Frontend queries GET /programs?sanatorium_id=…
  3. Filters programs where min_nights ≤ 7 ≤ max_nights
  4. Shows: "Your 7-night stay includes: Davolash (7 kun) — 19 services"
  5. Bookings only charge the room price; programs are bundled (free)

Wellness booking

  1. User picks a program directly (e.g. "Drop-in Yoga Class")
  2. Picks a date + participants
  3. Bookings charge program.price × guests
  4. No room involved at all
Frontend tip: on a sanatorium detail page, after the user picks dates you can call GET /programs?sanatorium_id=<id> and display only those with min_nights ≤ nights ≤ max_nights as "included treatment programs" — that's the only link between the two.

5. Sanatorium detail page

GET /api/v1/sanatoriums/{id}

Returns one sanatorium with embedded images and amenities. To populate the detail page completely you'll typically make 3 parallel calls:

Page sectionEndpoint
Header / about / amenities / mapGET /sanatoriums/{id}
Rooms tableGET /rooms?sanatorium_id={id}
Treatment programs sectionGET /programs?sanatorium_id={id}
ReviewsGET /sanatoriums/{id}/reviews
Extra beds (for booking widget)GET /extra-beds?sanatorium_id={id}

Search rooms by availability

When the user picks dates, hit:

GET /api/v1/rooms/search
  ?sanatorium_id=<id>
  &check_in=2026-06-01
  &check_out=2026-06-08
  &guests=2

Returns rooms with seats available for every date, plus computed final_price for the stay so cards can show "from $X".

7. Booking flow

POST /api/v1/bookings

Authenticated. The request body changes shape based on which kind of booking you're making — provide exactly one of room_category_id or program_id.

Room booking

POST /api/v1/bookings
Authorization: Bearer <token>

{
  "room_category_id": "16493749-…",
  "check_in":  "2026-06-01",
  "check_out": "2026-06-08",
  "guests": 2,
  "extra_beds": [
    { "config_id": "7a1f9073-…", "count": 1 }
  ]
}
  • Locks per-date inventory (FOR UPDATE)
  • Decrements units_available for each date
  • Computes final_price = Σ(per-night price) + Σ(extra_bed)
  • booking_type: "room"

Session booking

POST /api/v1/bookings
Authorization: Bearer <token>

{
  "program_id": "abc12345-…",
  "check_in":  "2026-06-01",
  "check_out": "2026-06-01",
  "guests": 2
}
  • Loads program (must be active + have a price)
  • Checks guests ≤ group_size_max
  • Computes final_price = program.price × guests
  • booking_type: "session"; check_out can equal check_in

Sample response (room)

{
  "id": "xxx",
  "code": "A3F9PB2K",
  "user_id": "<customer id>",
  "room_category_id": "16493749-…",
  "program_id": null,
  "booking_type": "room",
  "check_in":  "2026-06-01",
  "check_out": "2026-06-08",
  "guests": 2,
  "status": "confirmed",
  "final_price": "20300000.00",
  "currency": "UZS",
  "extra_beds": [
    { "id":"…","config_id":"…","name_snapshot":{ "ru":"Дети 4–10 лет" },

      "count":1,"total_price":"3500000.00" }

List my bookings

GET /api/v1/bookings — customers see their own; sanatorium admins see their venue's; super_admin sees all.

Cancel

PATCH /api/v1/bookings/{id}/cancel

  • Allowed by owner (own booking), sanatorium admin (their venue), or super_admin
  • Status must be pending or confirmed
  • Room: per-date units_available is restored
  • Session: status flip only

8. Pricing math

Room — per night

weekday   price = base_price          × (1 + markup/100) × (1 − discount/100)
weekend   price = base_price_weekend  × (1 + markup/100) × (1 − discount/100)

Friday + Saturday count as weekend. If the stay date falls inside a RoomPricePeriod, its base_price / base_price_weekend / discount_percent override the room defaults for that date (Humson Buloq has one for 8 Sep – 30 Dec 2025).

Stay total

booking.final_price
  = Σ(per-night price)
  + Σ(extra_bed.price_per_night × count × nights)

Session

booking.final_price = program.price × guests
Snapshots: once a booking is created, final_price, currency, extra-bed names and prices are frozen. Later changes to markup, room price, or extra bed config don't affect existing bookings.

UZS ↔ USD display

The backend stores native currency only (most rooms = UZS). For showing both, the frontend can call GET /api/v1/exchange-rates?pair=USD_UZS and compute the other side, or rely on the final_price_uzs / final_price_usd fields that /rooms/search returns.

9. Authentication

EndpointNotes
POST /auth/registerCreates a customer. Returns user + tokens.
POST /auth/login{"email": "...", "password": "..."} → { access_token, refresh_token, user }
POST /auth/refresh{"refresh_token": "..."} → new access token

Pass the access token as Authorization: Bearer <token> on every protected request.

Roles

RoleCan do
super_adminEverything: approve sanatoriums, set markup, manage amenity catalog

10. Errors

StatusTypical case
400Validation failure (e.g. check_out <= check_in, missing both room_category_id and program_id, both provided)
401Missing/invalid token on a protected route
403Authenticated but not allowed (e.g. admin editing a sanatorium they don't own)
404Resource not found or not visible to the caller (pending sanatorium for a customer)
409Conflict: no units available for a date; trying to cancel an already-cancelled booking
413Image upload exceeds 10 MB
415Unsupported image format (only JPEG/PNG/WebP)

Errors return:

{ "detail": "Human-readable message" }
